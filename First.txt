Inside libuv - callback queue, thread pool, event loop

callback queue - whenever v8 engine see aysnc operations it handles to libuv so when async code is executed and syncronous code
is still being executed by v8 so when the async code gets executed and sees v8 is still executing the code, means callstack is not 
empty then libuv will send the callback functions in callback queue, their are multiple type of async code like file read, request 
over server, timers etc, so their are different type of callback queue for async code 

event loop - constently checks if their is and code in callback queue and if the callstack is empty or not 

how event loop assigns priority between callback queue? like which one to execute first?

Phases 
Timer -> setTimeout,setInterval
pending callback -> executes I/O callbacks deferred to next loop iteration
pool -> incoming connections, data, fs, crypto, http.get
check -> setImmediate
close -> socket.on("close")

Internal loop 
process.nextTick()
promise callbacks

so whenever stack becomes empty then eventloop will check if their is any functions present in our callback and if their are
multiple then it will first do the internal loop checks if any functions that are present in process or promise queue then 
would follow the upper loop where it will go the phase wise but moving from one phase to another like if their is no callback 
function present in the timer queue then it would go to pool queue but before going it will check the internal loop again if
any callback function is present in any of the queue in the internal loop.

Thus in this way event loop goes executes in phases and sets the priority, although we are understanding this process 
but sometimes the exectution may work differently where we see timer exected later as it came late but pool exectued first
as it was executed and was present in the callback for exectution

Advance Concept - event loop waits at pool phase

Questions 
Lets suppose we are in pool phase , and thier is a callback in the queue (pool queue) , and when executed it calls promise and a timer
then will the event loop goes back to timer phase to execute it ? - No, it will move to next phase and before moving to next phase \
which is check phase it runs internal loop and their it checks the process queue and then promise queue where it sees a callback function
then it runs it and then move to all the phases then comes to timer phase again and sees the callback function and executes it

One full cycle is known as tick of event loop  

3 main components of libuv library are eventloop, callback queue, thread pool

Thread pool - as we know that js is single threaded synchronous language, and v8 sends the async operations to libuv
so v8 doesn't block the main thread, now for libuv inorder to execute the operations , it takes a thread from the thread pool
and then uses that thread to execute that async task 

whenever the thread complete the execution that it is now available in the thread pool

Now in nodejs the threadpool by default has only 4 threads and any other async task needs to be performed that it needs to 
wait for any thread from the thread pool to complete so that it can uses thread to execute it 

UV_THREADPOOL_SIZE = 4

we can change the thread size by using this line - process.env.UV_THREADPOOL_SIZE = 35

Now for this all task, libuv uses thread otherwise it can execute task normally too 
- fs, dsn.lookup, crypto, user specified input

Is Nodejs single threaded or multi-threaded ? 
Ans - depends like if their is any async code in program and it does the above operation than yeah node js is multi-threaded
but if their is simple sync code then it is single threaded
