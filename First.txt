that module by the means of require('file path')

Now when we use require to execute a module, it will execute that module but we can't use variables
or methods of that module inside another module, as the members in the module are protected and prevents from leaking
and logically thinking too it should be like that too if i have a module and inside their i use some variables
and do some operations around it and i use that module inside another module where same naming of variables took place then
their would be errors

so if we want to use fuctions/variables of another module we need to export and inside that module where we are using 
it the require will return all the exports of that module and we can import from the require 

var a =10;

}

module.exports ={ a:a,xyz:xyz}

const{a,xyz} = require('module path');

We know that js is single threaded syncronous language and it alone can't perform async, timer, api task
So Nodejs provides this powers of fetching data over server, reading file and timer by talking to OS
So inside Node thier is a library called libuv , which handles all the task of talking/performing with OS


libuv is a multi-platform C library. It was originally written for Node.js to provide a consistent API for asynchronous I/O operations across different operating systems
like Linux, macOS, and Windows.

It is a library inside Nodejs

-----
Now coming to async I/O operations in Node, it is being handled by libuv, but if all the async I/O gives output at the 
same time than the order of execution may look like this, here we can't say perfectly as it depends on which callback
gets gives output file even microseconds matter , so this is an upper manetal model to think off , no need to remember
The only thing to remember all the syncronous task gets executed first


1. Run all synchronous code
2. Run process.nextTick queue
3. Run Promise microtasks
4. Run timers (setTimeout)
5. Run I/O callbacks (fs, network)
6. Repeat


libuv also provides function like readFileSync() meaning whenever v8 encounters this line, it will handle to libuv
for execution as v8 doesnt have power to read/write file but will not move further in the program until the operations
is completed , this is also known as blocking the main thread , also a sync function will never take a callback, why?
A async function takes a callback and when the aysnc function completes it execution then it pushes the callback into the 
callstack for execution, but if the function is sync meaning thread  will wait until execution is completed then whats the need 
for a callback.

JUST FOR INFO

setTimeout, setInterval, clearTimeout, clearInterval, etc. are attached to the global object automatically by the runtime.

In Node.js, the global object is called global.

So internally, this works:

global.setTimeout(() => {
  console.log("Hi");
}, 1000);

But Node exposes those global functions directly into the global scope
