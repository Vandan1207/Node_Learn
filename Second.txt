Express.js
it is a web framework used to create webserver, it is easy to create fast, less redundant code 

why need for express?

Yes — you can absolutely build a web server using only Node's built-in http module.
Because once your app becomes larger, writing everything manually with the http module becomes repetitive, messy, and hard to maintain.

Imagine your app grows and you need:
Routing
JSON responses
Parsing request body
Middleware
Authentication
Logging
Error handling


Express is built on top of Node's http
Express is not replacing Node's http module.
Internally, Express still uses it.

Main things Express provides
1. Easier routing
2. Middleware system
3. Better request and response objects
4. URL parameters become easy
6. Built-in ecosystem of middleware


we can install express -> npm i express

package.json -> meta data about the project, what dependencies the project is using
example ->
    express:"^4.19.3"
    3-> patch
    19-> minor (backward compatible)
    4-> major (may have major changes, now their could be changes affecting our project)
    ^ -> upgrade to minor and patch changes automatically
    ~ -> patch updates only

Syntax for creating server using express

const express = require("express");
const app = express();

app.use("/ooo",(req,res)=>{
    res.send("listening on ooo path")
})

app.listen(7777,()=>{
    console.log("server is running")
})

Theory 
whenever a request is coming then express will check all the function and checks the matching url, and the one function
which send the response is known as handler function, and then express doesn't move forward, and all the function in between 
is known as middileware

middileware is a function that runs between a request and response and can be used in many ways like authorization, business logic etc..

As we know that order matters when defining routing / request handler , so how does middileware comes into play , is like 
on the top of all the methods we will define middileware, which is nothing but a request handler only, and it could for a 
perticular path or could be all the apis that hit , Syntax is somewhat like this

    next();
})

app.use() v/s app.all()

lets take a senario where the path is /users , so when we mention   this path in app.use(), all other apis with path /users/xyz , will be passed
through this app.use, but if we mention the same path in app.all, all the api request(get,put,post,patch) for that perticular path will only be 
passed through it

Few things about HTTP method, RestAPI


HTTP -  it is a protocol for communicating over the internet

HTTP header - (metadata) - key:value sent along with request & response
usecase - caching, Authentication, manage state 

standard - name start with X-custom header for your custom header

HTTP method - Basic set of opertation that can be used to interact with server
GET : retrive a message 
HEAD : No message body (response header only)
OPTIONS : what opertation are available
TRACE : loopback test (get some data)
DELETE : remove a resource
PUT : replace a resource
POST : interact with resource
PATCH : partial change in a resource 

HTTP Status Code 
1xx - Informational
2xx - Success
3xx - Redirection
4xx - Client Error
5xx - Server Error 



HTTPS - Their is an add on layer where in between the req and res, the data is encrypted 


RestAPI is a standard, which tells how an API should be made, it works on client-server architecture 
Their are 2 types of response, when it comes to client server architecture and it is also connected to how they are 
used for rendering, 1st - json where the data sent to client and based on it data will be either used for calculating, or 
rendering it on broswer, etc.. 2nd - html , where in the server based on the req we will sent the res as html, this is also 
known as SSR (Server side rendering) , it would only work if we know that the client is always a broswer, not another server
It uses HTTP, HTTP methods , but provides a standard/way on how to build using http method

URI (Uniform Resource Identifier)
